What is an API?

API stands for Application Programming Interface. In web development, an API is like a messenger between your code and other data sources or services. With JavaScript, you most often use APIs to:
  • Get dynamic data (news, weather, posts, user info)
  • Send data to a server (forms, uploads, etc.)
  • Power "live" features, search bars, or dashboards

Why Use Fetch?

JavaScript comes with the fetch() function for making HTTP requests. It works with APIs so your code can get or send data—straight from the browser, without page reloads. 

fetch('https://jsonplaceholder.typicode.com/todos/1')
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(err => console.error(err));
fetch() always returns a Promise. You use .then() and .catch() to wait for the response and handle success/failure.
You can use fetch for:
  • GET — Retrieve existing info (products, users, tasks)
  • POST — Send new data (create an item, send a message)
  • PUT, PATCH — Update something
  • DELETE — Remove an item

GET Requests: Fetching Data

The most basic fetch call uses just a URL. If you’re getting data (like a public feed, search, or sample for testing), use a simple GET request: 

fetch('https://jsonplaceholder.typicode.com/todos/3')
  .then(res => res.json())     // Parse the response as JSON
  .then(data => {
    console.log(data.title);  // Use the data
  });
The data you get is almost always in "JSON" format—a way to send objects and arrays!
Try practicing fetch with JSONPlaceholder, a free, fake API for learning.

Displaying Fetched Data in the Page

It's not enough to fetch data—you’ll want to show it on your own site. You’ll use DOM manipulation to put the fetched result in a list, table, or card. 

fetch('https://jsonplaceholder.typicode.com/users')
  .then(res => res.json())
  .then(users => {
    let html = users
      .map(user => '<li>' + user.name + '</li>')
      .join("");
    document.getElementById("userList").innerHTML = html;
  });

You get a JS array, so you can use map() and friends, just like with any local data—but now it’s live!

Sending Data: POST Requests

To send info to a server (like submitting a form or adding a to-do), give fetch a second argument—an options object. You almost always send JSON from JS. 

fetch('https://jsonplaceholder.typicode.com/posts', {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    title: "Learn Fetch",
    body: "APIs are fun",
    userId: 8
  })
})
.then(res => res.json())
.then(data => {
  console.log(data); // the new thing you created
});
Always set Content-Type: application/json when sending JSON data. Many real APIs will ignore your request if you don’t.
The body property must be a string. Use JSON.stringify() to convert your JS object into a data string.

Headers, Authentication, and Advanced Fetch

Most APIs require special headers for security, permissions, or tracking. Common uses:
  • Authorization: Include a user token or API key
  • Accept: Request a specific data format (application/json, application/xml, etc.)
  • X-Custom-Header: Extra info for advanced APIs

// Example: fetch with a token in the header
fetch('https://myapi.com/profile', {
  headers: {
    "Authorization": "Bearer my-secret-token-123",
    "Accept": "application/json"
  }
});
Never share secret keys or tokens in public code—keep them hidden or store them in a secure backend.
For protected APIs, you'll usually need to get a token by logging in, then include it with fetch for every future call.

Dealing With Errors

Not all API calls go smoothly! You might lose your network, or the server may refuse your data. Always handle errors so your users don’t get stuck or confused. 

fetch('https://jsonplaceholder.typicode.com/404stuff')
  .then(res => {
    if (!res.ok) throw new Error("Request failed: " + res.status);
    return res.json();
  })
  .then(data => console.log("Fetched:", data))
  .catch(err => console.error("Something went wrong:", err.message));
fetch does not fail on 404/500 by default; you must check res.ok to see if it worked.
Show helpful messages to users instead of raw JS errors.

Best Practices for APIs & Fetch

  • Always handle both .then and .catch when using fetch. Don’t assume success.
  • Show the user when data is loading, and handle “empty” or “error” states gracefully.
  • Never trust incoming data. Always validate or check it is what you expect before displaying!
  • Don’t leak API keys in your frontend code or to GitHub repos.
  • For repeated fetch patterns or large codebases, use wrapper functions to avoid repeating yourself.
  • Remember CORS: Modern browsers block some requests to protect users; only fetch from CORS-enabled APIs.
Learn more by exploring APIs with your browser tools (Network tab) to see requests, responses, and errors in real time.

Try each task before peeking. Building real projects means using fetch for almost every dynamic feature!
